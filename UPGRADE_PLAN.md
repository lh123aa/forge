## 二、迭代方向分析

### 2.1 功能优先级矩阵

| 功能 | 开发成本 | 用户价值 | 紧迫度 | 推荐 |
|------|----------|----------|---------|------|
| 工作流可视化 | 中 | ⭐⭐⭐⭐⭐ | 高 | ✅✅✅ |
| 错误恢复智能化 | 中 | ⭐⭐⭐⭐ | 高 | ✅✅✅ |
| 模板系统增强 | 低 | ⭐⭐⭐ | 中 | ✅✅ |
| 自我迭代实战化 | 高 | ⭐⭐⭐⭐ | 中 | ✅✅ |
| 知识图谱 | 高 | ⭐⭐⭐⭐ | 中 | ✅✅ |
| Skill 市场 | 中 | ⭐⭐⭐ | 低 | ✅ |
| 多 Agent | 极高 | ⭐⭐ | 暂缓 | ❌ |

### 2.2 不做多 Agent 的理由

| 原因 | 说明 |
|------|------|
| 单 Agent + Tools 足够强 | GPT-4/Claude 单 Agent 配合工具已能完成 90% 任务 |
| 复杂度爆炸 | 通信、状态同步、死锁调试难度指数级上升 |
| 行业未成熟 | LangChain Agents、AutoGPT 等多 Agent 实验大多停留 Demo 阶段 |
| 用户不为此付费 | 客户只关心结果，不关心内部是单 Agent 还是多 Agent |

### 2.3 什么时候多 Agent 有价值

- 角色分工明确（架构师 Agent + 开发者 Agent + 测试 Agent）
- 长程任务分解（规划 Agent 和执行 Agent 互不阻塞）
- 隐私敏感任务（不同 Agent 处理不同敏感数据）
- 容错需求高（单 Agent 失败需全部重做）

#### 3.2 错误恢复智能化

**优先级**: P0
**开发周期**: 约 2 周

**目标**:
- 错误自动分类（可重试 vs 需修复）
- 自适应重试间隔
- 提供修复建议

**技术方案**:
```
错误分类器
├── 可恢复错误
│   ├── 网络超时 → 指数退避重试
│   ├── 资源暂时不可用 → 等待后重试
│   └── 服务繁忙 → 限流重试
│
└── 不可恢复错误
    ├── 语法错误 → 生成修复建议
    ├── 依赖缺失 → 提示安装命令
    └── 业务逻辑错误 → 人工介入
```

**子任务**:
1. 扩展 SCAError 分类体系
2. 实现错误分类器（规则+轻量 ML）
3. 实现自适应重试策略
4. 实现修复建议生成器
5. 与 WorkflowExecutor 集成

**验收标准**:
- [ ] 错误自动分类准确率 > 80%
- [ ] 可恢复错误自动重试
- [ ] 不可恢复错误提供修复建议
- [ ] 重试间隔自适应调整
